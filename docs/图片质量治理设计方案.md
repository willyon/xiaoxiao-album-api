# 总览

这个相册系统项目是由我一个人进行全栈开发，目前处于开发测试阶段，所以设计方案时，不必顾虑数据问题。

已完整梳理前端 xiaoxiao-project 与后端（Node + Python AI 服务）现状，整理出当前持久化的图片元数据、人脸分析字段及处理流水线。结合行业通行做法（Apple Photos、Google Photos、Lightroom、Photoprism 等），给出围绕“相似图 / 模糊图智能清理”可直接落地的技术方案与实施步骤。

## 现有能力梳理（项目内）

### 数据层

- images 表已存储拍摄时间、位置信息、颜色主题、分辨率、文件大小、AI 分析结果（人脸数、人物数、表情/年龄/性别标签、主要人脸质量分、表情置信度等）。
- face_embeddings 表保存高质量人脸的 512 维特征向量。

### 处理流水线

1. 上传 → imageUploadIngestor 完成缩略图与基础入库。
2. imageMetaIngestor 提取 EXIF、生成高清图、补充元数据后入库，并将任务推送到 searchIndexQueue。
3. searchIndexIngestor.processFaceRecognition 调用 Python 服务 /analyze_person，把人物统计与人脸 embedding 写入数据库。

### Python AI 服务能力

- 可输出人物总数、面部质量分、表情/年龄/性别标签、faces 列表（含 embedding、quality_score）。
- 暂未生成图像整体的清晰度、相似度等指标。

### 前端能力

- Pinia imageStore 管理相册分页，PhotoInfoPanel 在开发模式下可展示人脸/人物信息。
- 上传流程已有重复文件检测（基于文件名+尺寸与 Redis 集合）。

## 行业方案调研要点

### 相似图 / 重复图检测

- 主流产品采用“多尺度指纹 + 语义向量”的双轨：快速 hash（dHash/pHash/aHash）用于大规模初筛，深度特征（CLIP、FACENET、Vision Transformer）或人脸 embedding 进行精细比对。
- 典型指标：cosine similarity > 0.92 视为重复；0.8~0.92 认定为相似，配合时间戳、拍摄连拍序列再判定。

### 模糊度 / 质量评估

- 常用无参考质量指标：Laplacian 方差、Tenengrad、SMD、Brenner、BLIINDS、BRISQUE。
- 谷歌 / 苹果倾向结合 Laplacian + ML 模型（如 NIMA、MANIQA）形成 0~1 质量分。
- 前端交互一般展示分组列表，默认勾选“低质量”分组，用户确认后批量删除 / 移动。

### 交互设计趋势

- 提供“智能清理中心”，按类别（重复、相似、模糊、截图、文档等）分卡片。
- 每组展示缩略图矩阵 + 主图标记 + 度量（相似度/清晰度）。
- 支持快速全选/反选、查看大图、仅保留最佳。
- 删除前给出影响提示，支持回收站恢复。

## 功能设计建议

### 总体架构

- 建议在 Python AI 服务 扩展图像质量评分、感知哈希、通用视觉 embedding，Node 侧负责调度与入库；保持 Node 服务轻量，避免重复实现矩阵运算。
- 新增一个 cleanup_insights 周期任务：
  - imageMeta 阶段或新增队列生成 hash/embedding/质量分。
  - Node 将指标入库，并维护“候选分组”。
  - 前端通过专用 API 拉取分组，支持批量操作。

### 数据设计（SQLite）

- images 表新增字段（均允许 NULL，保持幂等更新）：
  - perceptual_hash TEXT（pHash/dHash 16 字节 Hex）
  - aesthetic_score REAL（0~1，高分 = 更美观，可复用 NIMA/LAION aesthetic predictor）
  - sharpness_score REAL（Laplacian/Tenengrad 归一化得分）
  - similarity_cluster_id INTEGER（同一近似簇 ID）
  - duplicate_group_id INTEGER（高相似度判定的等价类）
  - 索引：idx_images_user_aesthetic, idx_images_user_sharpness, idx_images_user_dup_group.
- 新增 cleanup_groups 表（管理分组元信息）：
  - id, user_id, group_type (duplicate/similar/blurry/low_quality)
  - primary_image_id, score, member_count, created_at, updated_at.
- 新增 cleanup_group_members 表：
  - group_id, image_id, rank_score, is_recommended_keep (BOOLEAN), similarity (REAL), 主键 (group_id,image_id)。

### Python 服务扩展

#### 指纹与 embedding

- 生成多尺度 perceptual hash（如 pHash + wavelet hash），返回 Hex 字符串。
- 使用现有 face embedding（若有人脸）+ 通用模型（如 OpenCLIP ViT-B/32）生成 512/768 维向量，用于人物外的相似度判定。模型可通过 onnx 或 torch（若 CPU 可行）。

#### 模糊度与美学评分

- 实现 Laplacian 方差 + Tenengrad，归一化成 0~1 sharpness_score。
- 采用轻量模型（NIMA MobileNet/ONNX）或启用开源美学打分（LAION aesthetic predictor），返回 aesthetic_score。

#### 接口设计

- 确认新增 `/analyze_cleanup`，入参 image，可复用 convert_to_opencv，输出上述指标。
- 亦可考虑在 `/analyze_person` 输出附加字段（需注意兼容性）。

### Node 服务扩展

#### 队列设计

- 在 searchIndexQueue 处理完人脸后，于同一任务中调用 Python /analyze_cleanup，或独立新增 cleanupQueue（利于分阶段部署）。
- 已确认采纳方案：独立新增 cleanupQueue。

#### 业务服务

- cleanupService：负责将指标写入 images，构建/更新 cleanup_groups。
- 相似度计算策略：
  - 使用相同 perceptual_hash 直接归为 duplicate_group。
  - 基于 embedding 使用 cosine similarity + faiss/annoy/hnswlib 近邻检索（数据量大时）。初期可在 SQLite 内同步读取全部 embedding 计算（规模小），后续再外接 ANN。
  - 对同一 duplicate_group 按 aesthetic_score、sharpness_score、primary_face_quality 排序，标记 is_recommended_keep=true 的最佳图。
- 模糊图判定：设定阈值（如 sharpness_score < 0.2）+ aesthetic_score 低值 → 建立 blurry 组。支持按拍摄时间、是否无人脸等进一步过滤。

#### API 设计（REST）

- GET /cleanup/summary → 返回待清理类别及数量、节省空间估计。
- GET /cleanup/groups?type=duplicate&cursor=... → 分页返回分组、主图、推荐保留项、相似度。
- POST /cleanup/groups/:id/actions → 接受操作（删除所选或保留所选，所有分组需明确决策，不再提供忽略状态）。
- POST /cleanup/scan → 触发重新扫描（后台任务）。
- 所有删除动作走已有存储适配器 & 回收站策略（若暂无回收站，需要在方案中同步规划，防止误删）。

#### 调度

- 提供 cron/手动触发脚本（类似 batchEnqueueAIAnalysis.js）用于批量回填旧数据。

### 前端交互建议（Vue 3）

#### 清理入口

- 新增“智能清理”入口（主导航或设置面板）：
  - 顶部 Summary 卡片（重复图、相似图、模糊图、空间占用）。
  - Tab/Segment 分组显示不同类型。

#### 分组展示

- 每个分组展示：主图（推荐保留）、其他成员缩略图、相似度/清晰度标签、拍摄时间。
- 操作条：全部保留、删除非推荐、手动选择；需即时完成保留或删除的明确决策，不再提供“忽略”或“稍后处理”入口。
- 放大预览：支持放大预览（沿用已有 Lightbox 组件）。已确认继续统一使用 `PhotoPreview.vue`。

#### 状态与文案

- 进度感知：若后台扫描中，展示状态条 + ElProgress。
- Pinia 新 store（cleanupStore）：维护分组缓存、选择状态、批量操作。
- 文案：提示“删除后可在 X 天内通过回收站找回（若回收站上线）”。

### 运维与性能

- 指标计算可在上传时同步完成，也可离线批量回填（旧图像数量大时建议平滑迁移：先离线生成 → 写库，再逐步启用实时计算）。
- 哈希/embedding/质量分存表后，可按用户维度批量计算：避免跨用户数据泄漏。
- 延迟容忍度：第一次打开“智能清理”可触发后台扫描，提示“正在分析，稍后刷新”。
- 需要监控 Python 端模型加载耗时、内存占用；若 Apple Silicon/CPU 性能有限，可考虑使用 ONNXRuntime + FP16/INT8 模型。

## 实施步骤（建议迭代顺序）

1. 需求确认 & UX 线框：敲定分组界面、操作流程、误删保护策略。
2. 数据 schema 扩展：迁移脚本，添加新字段/表，更新 initTableModel.js。
3. Python 服务扩展：实现哈希/embedding/清晰度/美学评分接口，编写单元测试（针对常见格式）。
4. Node 后端：
   - 扩展 imageUnderstandingService 支持新接口；
   - 新增 cleanupService、cleanupController、API 路由；
   - 建立分组逻辑与写表流程；
   - CLI/脚本：reanalyzeCleanupMetrics.js 用于回填。
5. 前端：
   - 设计 UI 组件（分组卡片、批量操作条、进度条）；
   - Pinia store + API 封装；
   - 国际化文案。
6. 联调与验收：
   - 选取包含连拍/模糊照片的样本库验证分组准确度；
   - 检查数据库写入、删除流程、日志。
7. 性能优化（按需）：
   - 若 SQLite 近似查询瓶颈明显，引入 ANN 库或小型向量索引；
   - 缓存分组结果（Redis）以加速前端翻页。

## 风险与待确认

- 大规模历史数据回填时的算力与时间：是否需要离线批处理 + 进度监控。
- 回收站/恢复策略未明确：若当前删除即永久移除，需额外确认用户容忍度。
- Python 服务模型依赖（CLIP/NIMA 等）体积较大，需规划部署、缓存与更新方式。
- 若未来打算扩展到“截图/文档识别”，需在清理类型设计阶段预留扩展性。

## 综合说明

整体方案以现有架构为基础扩展，避免重复造轮子：Python 负责感知指标，Node 管理队列与数据持久化，前端提供直观的批量操作界面，可按阶段逐步落地。
